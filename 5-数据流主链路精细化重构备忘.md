## 3. 环节一：采集抓取

### 3.1 环节职责与边界

这是主链路的最前端环节，负责从配置指定的热榜源中抓取原始榜单内容，并把外部站点各自不同的返回格式，收敛为统一的“抓取层输出”。

这个环节只应该负责：

- 读取要抓哪些源
- 为每个源选择对应 handler
- 处理网络请求、重试、超时、代理
- 把每个源的返回结果规整成统一抓取结果
- 返回成功源、失败源、源名映射

这个环节不应该负责：

- 持久化到存储层
- 生成 `NewsData`
- 判断 report mode
- 计算新增标题
- 做 Selection / Insight / Render

也就是说，它的边界应该停在“抓到了什么”，不应该跨到“这些数据后面怎么用”。

### 3.2 当前功能模块

当前环节由以下模块组成。

#### A. 抓取源定义入口

- `config/config.yaml`
  - `platforms.enabled`
  - `platforms.sources`
- `newspulse/core/loader.py`
  - 负责把 YAML 中的 `platforms.sources` 读入运行时配置，落到 `config["PLATFORMS"]`
- `newspulse/context.py`
  - `AppContext.platforms` 暴露当前启用的平台列表

职责：

- 定义这次抓哪些热榜源
- 传给运行时编排层

#### B. 采集编排入口

- `newspulse/runner/news_runner.py`
  - `_crawl_data()`

职责：

- 从 `AppContext.platforms` 里取出平台
- 组装抓取用的 `ids`
- 调用 `DataFetcher.crawl_websites()`
- 接收抓取结果，交给后续环节

它当前仍是“抓取环节”和“后续归一化/入库环节”的交界点。

#### C. 抓取协调器

- `newspulse/crawler/fetcher.py`
  - `DataFetcher.fetch_data()`
  - `DataFetcher.crawl_websites()`

职责：

- 串行遍历每个 source
- 处理重试和随机等待
- 调用具体 source handler
- 汇总结果为统一抓取层输出

#### D. Source 注册中心

- `newspulse/crawler/sources/registry.py`
  - `SOURCE_REGISTRY`
  - `get_source_handler()`

职责：

- 把 `source_id` 映射到具体抓取函数
- 提供“source id -> handler”的分发机制

当前它是抓取环节里最核心的“源路由层”。

#### E. Source 基础客户端与通用工具

- `newspulse/crawler/sources/base.py`
  - `SourceClient`
  - `SourceItem`
  - HTML / RSS / URL / hash 等基础工具
- `newspulse/crawler/sources/common.py`
  - 文本清洗、通用 item 构造、feed 辅助函数

职责：

- 封装 requests session、代理、headers、timeout
- 给 source handler 提供统一网络访问能力
- 提供抓取后最小归一化对象 `SourceItem`

#### F. 各垂直 source handler 实现

- `newspulse/crawler/sources/mainland.py`
- `newspulse/crawler/sources/finance.py`
- `newspulse/crawler/sources/tech.py`
- `newspulse/crawler/sources/misc.py`

职责：

- 面向单一站点抓取
- 解析 JSON / HTML / RSS / 页面结构
- 输出 `list[SourceItem]`

这些 handler 才是真正“贴着外部网站协议”的地方。

### 3.3 当前输入数据

当前环节的输入主要有三类。

#### 输入 1：平台配置

来源：

- `config/config.yaml`
- `newspulse/core/loader.py`
- `newspulse/context.py`

当前运行时形状大致为：

```python
[
    {"id": "thepaper", "name": "澎湃"},
    {"id": "tencent-hot", "name": "腾讯热榜"},
    {"id": "cls-hot", "name": "财联社热榜"},
    {"id": "wallstreetcn-hot", "name": "华尔街见闻热榜"},
    {"id": "github-trending-today", "name": "GitHub Trending"},
    {"id": "hackernews", "name": "Hacker News"},
    {"id": "juejin", "name": "掘金"},
]
```

#### 输入 2：抓取运行参数

来源：

- `newspulse/runner/news_runner.py`
- `newspulse/context.py`

包括：

- `request_interval`
- `proxy_url`
- source 列表
- source alias / name

#### 输入 3：外部站点响应

来源：

- 各 source handler 访问的站点 API、HTML 页面、RSS feed

当前这是最不稳定的一层输入，不同站点的返回格式完全不同，因此都要在 handler 内部消化。

### 3.4 当前输出数据

当前这个环节最终输出 3 个对象：

1. `results`
2. `id_to_name`
3. `failed_ids`

其中 `results` 目前还是一个偏旧的 dict 形状：

```python
{
    "hackernews": {
        "OpenAI launches a new coding agent": {
            "ranks": [1],
            "url": "https://news.ycombinator.com/item?id=1",
            "mobileUrl": ""
        }
    }
}
```

`id_to_name` 形状：

```python
{
    "hackernews": "Hacker News",
    "github-trending-today": "GitHub Trending",
}
```

`failed_ids` 形状：

```python
["thepaper", "cls-hot"]
```

这一点很重要：

- source handler 的直接输出其实已经是 `list[SourceItem]`
- 但 `DataFetcher.crawl_websites()` 又把它重新折回了旧 dict 形状

这正是当前采集抓取环节最大的技术债之一。

### 3.5 数据从进来到出去的完整处理链路

下面按真实执行顺序说明。

#### 第 1 步：加载启用的平台列表

- `config/config.yaml` 中定义 `platforms.sources`
- `newspulse/core/loader.py` 把它写入 `config["PLATFORMS"]`
- `newspulse/context.py` 通过 `AppContext.platforms` 暴露给运行时

此时数据还是“配置对象列表”。

#### 第 2 步：运行编排层组装抓取请求

- `newspulse/runner/news_runner.py::_crawl_data()`

这里会把平台列表转换成：

- `source_id`
- 或 `(source_id, alias)` 二元组

得到一个给 `DataFetcher` 使用的 `ids` 列表。

#### 第 3 步：抓取协调器遍历 source

- `newspulse/crawler/fetcher.py::crawl_websites()`

行为包括：

- 遍历 `ids_list`
- 为每个 source 保留 `id_to_name`
- 调用 `fetch_data()`
- 相邻 source 之间做随机等待

此时系统开始真正触网。

#### 第 4 步：单个 source 执行抓取

- `newspulse/crawler/fetcher.py::fetch_data()`

行为包括：

- 根据 `source_id` 调 `get_source_handler()`
- 从 `SOURCE_REGISTRY` 中找到 handler
- 执行 handler
- 失败时进行重试

这一层的本质是“source 调度 + retry policy”。

#### 第 5 步：source handler 访问外部站点并解析

对应文件包括：

- `newspulse/crawler/sources/mainland.py`
- `newspulse/crawler/sources/finance.py`
- `newspulse/crawler/sources/tech.py`
- `newspulse/crawler/sources/misc.py`

这里会使用：

- `SourceClient.get_json()`
- `SourceClient.get_text()`
- `SourceClient.get_soup()`
- `SourceClient.get_feed()`

完成：

- 发请求
- 解析响应
- 提取标题 / URL / mobile URL
- 构造成 `SourceItem`

到这一步，单个站点的数据已经被规整成统一对象：

```python
SourceItem(
    title="OpenAI launches a new coding agent",
    url="https://news.ycombinator.com/item?id=1",
    mobile_url=""
)
```

#### 第 6 步：抓取协调器把 `SourceItem[]` 再折成旧汇总结果

- `newspulse/crawler/fetcher.py::crawl_websites()`

它会把 `list[SourceItem]` 转成：

- 以 `source_id` 为第一层 key
- 以 `title` 为第二层 key
- 值中记录 `ranks / url / mobileUrl`

并在同标题多次出现时，把排名累计到 `ranks` 列表。

这一层同时还会：

- 汇总 `id_to_name`
- 汇总 `failed_ids`

#### 第 7 步：返回给运行编排层

- `results`
- `id_to_name`
- `failed_ids`

返回点：

- `newspulse/runner/news_runner.py::_crawl_data()`

到这里，采集抓取环节结束。

后续下一环节“原始结果归一化 / 入库”会继续处理这些输出，把它们转换为 `NewsData` 并落库。

### 3.6 当前耦合点 / 脏点 / 旧形状残留

这一环节当前主要有 6 个问题。

#### 问题 1：抓取输出仍然回落到旧 dict 结构

source handler 已经产出 `SourceItem`，但 `DataFetcher.crawl_websites()` 又把它折回：

- `results[source_id][title]["ranks"]`
- `url`
- `mobileUrl`

这会让下游继续依赖旧结构，而不是直接吃原生抓取模型。

#### 问题 2：抓取协调器同时承担了“抓取”和“汇总归一化”

`DataFetcher` 现在既做：

- source 调度
- retry
- wait interval
- 聚合
- title 去重
- rank list 拼装

职责偏多。

#### 问题 3：source registry 是平铺硬编码表

`SOURCE_REGISTRY` 现在是一个大字典，虽然简单直接，但后续：

- source 元信息
- source 分类
- source 能力标签
- source 限流策略

都还没有结构化承载位置。

#### 问题 4：抓取层与运行编排层耦合较深

`NewsRunner._crawl_data()` 还在自己拼 source id/alias，并直接感知后续入库逻辑。

这说明“抓取任务定义”还没有独立成干净对象。

#### 问题 5：抓取失败信息过粗

当前 `failed_ids` 只有 source id，没有结构化失败原因、异常类型、响应信息。

这会限制：

- 观测性
- 重试诊断
- 后续失败源展示

#### 问题 6：缺少独立的采集层契约

当前采集环节没有像 `HotlistSnapshot`、`SelectionResult` 那样清晰的 stage contract。

这导致采集输出在进入存储层前，仍然是“临时拼出来的结构”。

### 3.7 本环节的精细化重构目标

后续这个环节应收口为：

```text
平台定义
-> 抓取任务
-> source handler
-> 原生抓取结果模型
```

目标包括：

1. 定义采集层原生输出模型，不再把 `SourceItem[]` 折回旧 dict
2. 把“单源抓取结果”“整批抓取结果”“失败结果”结构化
3. 把 retry / interval / source dispatch 与结果汇总拆开
4. 让 `runner` 不再感知抓取层内部拼装细节
5. 为下一环节“原始结果归一化 / 入库”提供稳定输入契约

建议未来目标形状至少包括：

- `FetchedSourceItem`
- `FetchedSourceBatch`
- `FetchFailure`
- `CrawlBatchResult`

### 3.8 建议拆分任务

建议把本环节再拆成以下子任务逐个做。

#### 任务 1：定义原生采集结果模型

目标：

- 替代 `results / id_to_name / failed_ids` 三件套

#### 任务 2：拆分 source dispatch 与 batch aggregation

目标：

- 让“抓单个源”和“汇总整批结果”成为两个清晰模块

#### 任务 3：把失败信息结构化

目标：

- 失败不再只有 source id
- 至少要包含 source_id、source_name、reason、exception_type

#### 任务 4：收紧 source registry 元信息

目标：

- 给每个 source 附带更稳定的定义对象，而不是只存函数指针

#### 任务 5：让 `runner` 只消费采集层契约

目标：

- `NewsRunner` 不再处理采集层内部数据拼接

### 3.9 验收标准

本环节完成时，应满足：

- 抓取层存在明确原生输出模型
- `DataFetcher` 不再输出旧 `results[source_id][title]` dict 结构
- source handler 统一输出稳定 item 契约
- 失败信息具备结构化字段
- `runner` 只消费“整批抓取结果对象”，而不是三件套临时结构
- 下一个环节“原始结果归一化 / 入库”直接消费新的采集层输出

### 3.10 本阶段实施范围

本阶段明确要做以下 5 件事，同时解决前面已经记录的主要问题：

1. 定义原生采集结果模型
2. 拆开 source dispatch 和 batch aggregation
3. 结构化失败信息
4. 收紧 source registry 元信息
5. 让 `runner` 只消费采集层契约，不再碰临时三件套

补充约束：

- 本阶段的重点是“采集层输出契约原生化”
- 不在本阶段重写整个 storage 仓储实现
- 但允许调整采集层到 storage 入口之间的转换函数签名
- 不允许继续让主链路生产代码依赖 `results / id_to_name / failed_ids` 三件套

### 3.11 代码现状分析结论

结合当前代码，采集环节的问题不是单点问题，而是 4 个问题缠在一起。

#### 结论 1：最核心的问题是“source handler 已原生化，但 fetcher 又把它降级了”

现在每个 source handler 的输出已经是：

- `SourceItem(title, url, mobile_url)`

这是一个很好的最小统一对象。

但 `newspulse/crawler/fetcher.py::crawl_websites()` 又把它折回：

- `results[source_id][title]["ranks"]`
- `url`
- `mobileUrl`

所以目前真正该改的不是 handler，而是：

- batch 输出契约
- fetcher 的职责边界
- runner 与 storage 的接缝

#### 结论 2：`crawl_websites()` 同时承担了三层职责

当前 `DataFetcher.crawl_websites()` 同时做了：

1. source 遍历与调度
2. source 抓取失败重试
3. 旧结构聚合与失败列表汇总

这导致它很难继续演进。

更合理的结构应该是：

```text
Source dispatch
-> per-source fetch result
-> batch result assembly
```

其中：

- dispatch 负责“调谁”
- fetch 负责“抓什么”
- batch assembly 负责“整批结果怎么组织”

#### 结论 3：结构化失败信息不能只在 crawler 内部改，必须考虑后续边界

当前失败信息只有：

- `failed_ids: list[str]`

而下游真正依赖失败信息的地方包括：

- `newspulse/storage/base.py`
- `newspulse/storage/repos/news.py`
- `newspulse/workflow/snapshot/service.py`
- `newspulse/workflow/render/html_page.py`
- `newspulse/workflow/render/notification_content.py`

因此合理方案不是只在 crawler 里加个异常对象就结束，而是：

- crawler 先输出结构化失败对象
- 采集到 storage 的转换入口至少先能理解结构化失败
- 之后由下一环节继续把失败信息完整存储和透传

#### 结论 4：source registry 现在的问题不是“能不能用”，而是“信息密度太低”

当前 registry 只保存：

- `source_id -> handler`

这足够运行，但不够支撑后续重构。

本阶段更合理的方向是让 registry 至少能表达：

- canonical id
- aliases
- default display name
- category / group
- handler

这样：

- 配置里可以继续写 alias
- 运行时可以统一到 canonical id
- 失败信息、日志、后续统计都可以更稳定

### 3.12 合理方案：目标结构

建议把采集环节收口成下面这条链路：

```text
平台配置
-> CrawlSourceSpec
-> SourceDefinition
-> SourceDispatcher
-> SourceFetchResult / SourceFetchFailure
-> CrawlBatchResult
-> convert_crawl_batch_to_news_data(...)
```

这里有几个关键设计决定。

#### 设计决定 A：保留 `SourceItem` 作为“handler 最小输出”，但不再让它直接充当整批抓取契约

原因：

- handler 只关心把站点数据解析成有序条目列表
- `SourceItem` 目前已经足够轻量稳定
- 真正需要升级的是“单源批次结果”和“整批抓取结果”

所以：

- `SourceItem` 可以继续保留在 `newspulse/crawler/sources/base.py`
- 但 batch 级别新增单独 dataclass

#### 设计决定 B：标题去重与 `ranks` 聚合从采集层退出

当前 `crawl_websites()` 会在抓取层就把同标题合并并累计 `ranks`

这不够合理，因为这其实已经是“原始结果归一化”的职责，应该属于下一环节。

因此本阶段建议：

- 采集层只保留“有序原始条目列表”
- 同标题聚合、`ranks` 生成、`rank=第一名次` 推导，交给“原始结果归一化 / 入库”环节

这会让两个环节边界更清楚。

#### 设计决定 C：失败对象先在采集层结构化，storage 入口先做消费

建议的新失败模型至少包含：

- `source_id`
- `source_name`
- `exception_type`
- `message`
- `attempts`
- `retryable`

如果本阶段不想立刻重写底层数据库表，可以先做到：

- 采集层输出结构化失败
- `convert_crawl_batch_to_news_data(...)` 暂时只把 `source_id` / `source_name` 映射进旧 `NewsData`

但这只能是“边界过渡”，不能再让 `runner` 继续拿三件套。

#### 设计决定 D：`runner` 的 `_crawl_data()` 改为只返回一个对象

当前：

```python
results, id_to_name, failed_ids = self.data_fetcher.crawl_websites(...)
```

应改成：

```python
crawl_batch = self.data_fetcher.crawl(...)
```

`runner` 只应做：

- 发起采集
- 把 `crawl_batch` 交给存储入口转换函数
- 不再理解内部三件套细节

### 3.13 推荐的数据模型方案

建议新增 `newspulse/crawler/models.py`，定义以下对象。

#### 1. `CrawlSourceSpec`

表示一次运行里“要抓哪个源”。

建议字段：

- `source_id: str`
- `source_name: str`

作用：

- 承接 `config["PLATFORMS"]`
- 作为 runner 传给 crawler 的稳定输入模型

#### 2. `SourceDefinition`

表示 registry 中的一个源定义。

建议字段：

- `canonical_id: str`
- `default_name: str`
- `aliases: tuple[str, ...]`
- `category: str`
- `handler: SourceHandler`

作用：

- 统一 alias
- 解决 registry 元信息过弱的问题

#### 3. `FetchedSourceItem`

## 6. 环节四：Selection

### 6.1 环节职责与边界

Selection 是主链路里第一个真正“做判断”的环节。

它的职责不是采集，也不是写报告，而是：

- 从 `HotlistSnapshot` 里判断“哪些新闻值得进入后续主链路”
- 把入选新闻按主题聚合成 `SelectionGroup`
- 产出后续 `Insight / Report Assembly / Localization / Render / Delivery` 可直接消费的 `SelectionResult`

这个环节应该只关心“选什么、为什么选、按什么主题输出”，不应该负责：

- 回头改写采集结果
- 直接生成 Insight 文案
- 直接拼 HTML / 通知文本
- 把主链路输出重新折回旧 `report_data` 结构

因此，Selection 的天然边界应该是：

- 输入：`HotlistSnapshot`
- 输出：`SelectionResult`
- 中间允许有多层筛选与判定，但这些层都必须服务于同一个原生 stage contract

### 6.2 当前功能模块

当前第四环节已经有了原生 stage contract，但内部仍然明显分裂成“关键词宇宙”和“AI 宇宙”两条链路。

#### A. Stage 入口与调度

- `newspulse/context.py`
  - `create_selection_service()`
  - `build_selection_options()`
  - `run_selection_stage()`
- `newspulse/workflow/selection/service.py`
  - `SelectionService.run()`

职责：

- 构造 `SelectionOptions`
- 根据 `strategy` 选择 keyword 或 ai
- 在 AI 失败时做 fallback
- 统一补齐 `selected_new_items`

#### B. 关键词筛选栈

- `newspulse/workflow/selection/frequency.py`
  - 频率词文件解析
  - typed keyword rule set 装载
- `newspulse/workflow/selection/models.py`
  - `KeywordToken`
  - `KeywordRuleGroup`
  - `KeywordRuleSet`
  - `KeywordGroupBucket`
- `newspulse/workflow/selection/keyword.py`
  - `KeywordSelectionStrategy`

职责：

- 读取 `frequency_words.txt`
- 做全局过滤、组级过滤、必选词匹配
- 把命中的标题聚合为 `SelectionGroup`

#### C. AI 筛选栈

- `newspulse/workflow/selection/ai.py`
  - 当前是 AI selection orchestrator
- `newspulse/workflow/selection/ai_tags.py`
  - 从 `ai_interests.txt` 提取 tag
  - 管理 active tag / analyzed ids / prompt hash
- `newspulse/workflow/selection/ai_classifier.py`
  - 构造 batch
  - 整批调用 LLM 分类
  - 递归拆批处理失败
- `newspulse/workflow/selection/ai_results.py`
  - 将 tag 分类结果投影成 `SelectionResult`

职责：

- 从自然语言兴趣文件生成主题标签
- 把整批标题交给 LLM 做主题归类
- 读取 / 保存 AI 分类结果
- 输出 AI 主题组

#### D. 审阅与验证辅助

- `newspulse/workflow/selection/review.py`
- `newspulse/workflow/selection/audit.py`

职责：

- 导出 stage4 snapshot / keyword / ai 产物
- 生成人工审阅文档
- 支撑 Selection 改造过程中的质量回看

### 6.3 当前输入数据

当前 Selection 环节实际消耗 4 类输入。

#### 输入 1：标准化快照

来源：

- `newspulse/workflow/snapshot/*`
- `newspulse/workflow/shared/contracts.py`

当前核心输入对象是 `HotlistSnapshot`，其中最关键的是：

- `items`
- `new_items`
- `failed_sources`
- `standalone_sections`

其中真正参与 Selection 判断的主输入仍然是 `snapshot.items`。

#### 输入 2：关键词规则

来源：

- `config/frequency_words.txt`
- `newspulse/workflow/selection/frequency.py`

当前已经比旧版更好，内部不再只是散乱字符串，而是：

- `KeywordRuleSet`
- `KeywordRuleGroup`
- `KeywordToken`

这部分是目前最稳定的一层 Selection 输入。

#### 输入 3：兴趣描述与 AI runtime

来源：

- `config/ai_interests.txt`
- `.env`
- `config/config.yaml`
- `newspulse/context.py`

当前 AI selection 的主题来源并不是 typed topic catalog，而是：

- 一份自然语言兴趣文件
- 运行时再通过 LLM 抽取 tag

这意味着主题定义在运行前仍然是不稳定、非结构化的。

#### 输入 4：历史分类状态

来源：

- `storage/ai_filter_*`
- `newspulse/storage/repos/ai_filter.py`

当前 AI selection 为了减少重复分类，会额外依赖：

- active tags
- active filter results
- analyzed news ids
- interests hash

这些并不是 Selection 的主输入，但已经深度影响 AI 分支的执行路径。

### 6.4 当前输出数据

当前第四环节对主链路真正输出的是 `SelectionResult`。

形状大致为：

```python
SelectionResult(
    strategy="keyword" | "ai",
    groups=[SelectionGroup(...)],
    selected_items=[HotlistItem(...)],
    selected_new_items=[HotlistItem(...)],
    total_candidates=128,
    total_selected=44,
    diagnostics={...},
)
```

这说明一件很重要的事：

- Selection 的“对外 contract”其实已经基本正确
- 真正需要重构的是内部判断链路，而不是再次推翻 `SelectionResult`

本轮第四环节重构，应该保留这个对外 contract，重点收口内部到统一的五层筛选管线。

### 6.5 数据从进来到出去的完整处理链路

#### 当前 keyword 链路

1. `AppContext.run_selection_stage()` 构造 `SelectionOptions(strategy="keyword")`
2. `SelectionService.run()` 路由到 `KeywordSelectionStrategy`
3. `frequency.py` 读取 `frequency_words.txt`
4. `keyword.py` 遍历 `snapshot.items`
5. 按规则命中标题并分配到 bucket
6. 生成 `SelectionGroup[]`
7. 扁平化为 `selected_items`
8. `SelectionService` 再统一计算 `selected_new_items`

这条链路的问题不在“能不能跑”，而在于它只懂字面命中，不懂语义。

#### 当前 AI 链路

1. `AppContext.run_selection_stage()` 构造 `SelectionOptions(strategy="ai")`
2. `SelectionService.run()` 路由到 `AISelectionStrategy`
3. `ai_tags.py` 读取 `ai_interests.txt`
4. LLM 提取 tag，并与历史 active tags / prompt hash 对齐
5. `ai_classifier.py` 将所有候选标题按 batch 扔给 LLM 做分类
6. 分类结果保存到 `ai_filter` 存储表
7. `ai_results.py` 将结果重新投影为 `SelectionGroup[]`
8. `SelectionService` 统一补 `selected_new_items`

这条链路的问题也不在“能不能出结果”，而在于：

- 它把“主题发现、候选召回、最终判断”全部压进了 LLM 一步
- 当快照里有 100+ 标题时，prompt 上下文过长且噪声过大
- 分类理由、候选来源、置信度证据没有被清晰分层保留

### 6.6 当前耦合点 / 脏点 / 旧形状残留

当前第四环节的核心问题已经比较明确。

#### 问题 1：关键词筛选与 AI 筛选仍是两套平行系统

- keyword 分支读 `KeywordRuleSet`
- ai 分支读 `ai_interests.txt + active tags`
- 两条链路只在最后同构到 `SelectionResult`

这意味着：

- 前面的候选召回逻辑不能复用
- 中间证据不能统一表达
- 难以演进为“keyword -> semantic -> classifier -> llm”的多层管线

#### 问题 2：AI 仍在做“大 batch 全量主分类”

当前 AI 分支会把大量标题直接交给 LLM 分类。

这会带来：

- prompt 太长
- 分类边界受噪声影响
- source 分布不均时容易被某个平台带偏
- 成本与延迟都偏高

它更像“把整个 Selection 外包给 LLM”，而不是“让 LLM 只处理真正模糊的样本”。

#### 问题 3：缺少语义召回层

现在 keyword 只能做字面命中，AI 又直接做最终分类，中间没有：

- embedding 召回
- 近义表达补全
- 多语言 / 中英混合标题的相似度判断
- 模糊主题的候选收窄

这正是目前漏召回与 LLM 压力过大的共同根因。

#### 问题 4：AI 主题仍是非结构化的

`ai_interests.txt` 目前是自然语言描述，运行时临时抽 tags。

这样的问题是：

- 主题定义不可直接复用
- 难以给 semantic recall 提供稳定 query
- 难以给 classifier 提供固定 label space
- 难以保留 topic-level metadata（alias、seed、negative、description、priority）

#### 问题 5：AI 分支仍直接耦合具体存储细节

当前 `ai_tags.py / ai_classifier.py` 直接使用 `storage_manager` 上的 `ai_filter_*` 方法。

这会导致：

- Selection 的内部算法与旧存储细节绑得太紧
- 后面想换 classifier / semantic memory 时改动面偏大
- stage 内部的“状态持久化”与“筛选逻辑”没有清晰分层

#### 问题 6：输出里缺少跨层证据

虽然最终有 `SelectionGroup`，但当前 `SelectionResult` 里还没有系统性保留：

- 该条新闻是被 keyword 命中还是 semantic 召回
- semantic score 是多少
- classifier / llm 各自的置信度是多少
- 最终为什么归到这个 topic

这会直接削弱后续审阅、调参和回归验证能力。

### 6.7 精细化重构目标

第四环节的目标不是“再做一个新 AI 筛选器”，而是把 Selection 真正收口为统一的五层原生管线。

固定目标如下：

#### 目标 1：保留现有 `SelectionResult` 作为唯一对外输出

对外仍然只输出：

- `SelectionGroup[]`
- `selected_items`
- `selected_new_items`
- `diagnostics`

不要让后续环节感知 Selection 内部到底用了 keyword、semantic、classifier 还是 llm。

#### 目标 2：把内部收敛为 5 层

固定链路：

1. `keywords` 筛选层
2. `semantic recall` 语义召回层
3. `lightweight classifier` 轻量分类层
4. `llm arbitration` LLM 仲裁层
5. `final projection` 最终输出层

以后所有 Selection 能力演进，都只允许落在这 5 层之内。

#### 目标 3：第二层先用 `.env` 中与 LLM 同源的 embedding 模型

本轮明确采用：

- LLM 仍走当前 `AI_FILTER_MODEL`
- embedding 走 `.env` 中单独配置的 `EMB_MODEL`
- `API_KEY / API_BASE` 复用当前同源 provider

也就是说，本轮不引入本地 embedding 大模型，不引入额外 provider，先把“同源 embedding + 语义召回”跑通。

#### 目标 4：重排序先预留接口，不在本轮强行启用

第二层只先完成：

- embedding 向量化
- 相似度召回
- top-k 候选集输出

同时预留 `reranker` 接口，但默认关闭。

这样可以先把主链路稳定下来，后续再决定接：

- provider 同源 rerank
- 本地 reranker
- 其他更强的精排策略

#### 目标 5：第三层先做“可落地、可训练、可关闭”的轻量分类

第三层不能一开始就依赖重型本地训练。

结合当前机器条件：

- GPU：RTX 5070 8GB

结论应该是：

- 小模型 / 小 batch 的轻量分类训练是可行的
- 但不适合把大体量 encoder 微调、语义主召回和复杂训练全部压到本机上

因此第三层的合理方向是：

- 先抽象 classifier 接口
- 先沉淀训练样本与评估集
- 默认支持 `noop` / baseline classifier
- 真正启用训练时优先选“小模型、小 batch、短文本”的方案

换句话说，本机性能足够支撑“轻量分类层”，但不应该把它设计成 Selection 的硬依赖。

#### 目标 6：让第四层 LLM 只做仲裁，不再做全量主分类

LLM 只应该处理：

- 多 topic 接近的边界样本
- keyword 未命中但 semantic 高相似的样本
- classifier 低置信或冲突样本
- 需要最终剔除噪声的样本

而不应该再承担“把 100+ 标题从头到尾全部分类一遍”的工作。

### 6.8 建议拆分任务（第四环节实施方案 v2）

下面按建议落地顺序展开。

#### 任务 1：先定义 Selection 原生中间契约

新增一组仅属于第四环节内部的原生模型，建议放在：

- `newspulse/workflow/selection/models.py`
- 或拆成 `topics.py / pipeline_models.py`

建议至少定义：

- `SelectionTopic`
  - `topic_id`
  - `label`
  - `description`
  - `priority`
  - `seed_keywords`
  - `negative_keywords`
  - `source`（来自 interests / static config / future training）
- `SelectionCandidate`
  - `news_item`
  - `topic_id`
  - `source_layers`
  - `scores`
  - `evidence`
- `SelectionDecision`
  - `news_item_id`
  - `final_topic_id`
  - `decision_layer`
  - `decision_reason`
  - `scores`

这一步是整个重构的根。

没有这层 typed contract，后面的 semantic / classifier / llm 仍然会各做各的。

#### 任务 2：把主题定义从“纯自然语言文件”收紧为可复用 topic catalog

不改 `frequency_words.txt` 的职责，也不要求马上废弃 `ai_interests.txt`。

本轮建议：

- `frequency_words.txt` 继续只服务第一层 keyword
- `ai_interests.txt` 继续作为主题来源文本
- 但运行时先把 `ai_interests.txt` 解析 / 抽取为 `SelectionTopic[]`

也就是说，后续第二、三、四层都不再直接面对一整段 interests 原文，而是统一面对 topic catalog。

这样可以把：

- semantic recall 的 query
- classifier 的 label space
- llm arbitration 的候选主题

全部建立在同一套主题对象上。

#### 任务 3：新增 Embedding runtime 与语义召回层

建议新增：

- `newspulse/workflow/shared/ai_runtime/embedding.py`
- `newspulse/workflow/selection/semantic.py`

职责拆分为：

- embedding client：只负责请求向量
- semantic selector：只负责构造文本、算相似度、产出候选集

配置策略：

- 默认从 `.env` 读取 `EMB_MODEL`
- `API_KEY / API_BASE / timeout` 默认继承 `AI_FILTER_MODEL`
- 支持后续在 `config.yaml` 中显式覆盖

第一版召回文本建议只使用：

- `title`
- `source_name`

暂时不要把正文抓取前置依赖引进第四环节。

第二层输出应该是：

- 每条新闻的 top-k topic 候选
- 对应 semantic score
- 一个可选的“直接通过阈值”

#### 任务 4：预留 reranker 接口，但默认关闭

建议新增一个很薄的接口，例如：

- `SelectionReranker`
- `NoopSelectionReranker`

第二层先输出初召回候选，若后续开启 rerank，则在这里重排。

本轮不要求真的接精排模型，但要把调用点留好，避免以后再改 Selection 主骨架。

#### 任务 5：第三层做轻量分类接口与训练数据沉淀

本轮第三层建议拆成两部分：

1. 运行时接口
   - `SelectionClassifier`
   - `NoopSelectionClassifier`
   - `ClassifierPrediction`
2. 训练 / 标注资产沉淀
   - 从 stage4 审阅结果导出训练样本
   - 形成 topic -> title 的监督数据

结合当前机器条件，执行建议是：

- 第一版不把重训练设为上线前提
- baseline 可先 CPU 跑通
- 若后续启用小型 classifier 训练，再单独引入训练依赖与脚本

这能保证：

- 先完成 Selection 主链路改造
- 不因为本机训练链路把本轮 scope 拉爆

#### 任务 6：重写第四层 LLM，为“仲裁器”而不是“总分类器”

建议把现有 `AISelectionStrategy` 的职责收窄。

未来它不应直接面对整批 snapshot items，而应只接收：

- semantic recall 的候选集
- classifier 的预测结果
- 少量边界样本

LLM 输入应变成：

- 单条新闻或小 batch 新闻
- 每条新闻携带少量候选 topic
- 每个 topic 附带 description / priority / 已有证据

这样可以显著缩短 prompt，减少 source 噪声，并让 LLM 只做“裁决”。

#### 任务 7：新增统一聚合器，输出唯一 `SelectionResult`

建议新增一个 stage 内部聚合器，例如：

- `selection/pipeline.py`
- 或 `selection/projector.py`

负责把 1-4 层的结果统一折叠成：

- `SelectionGroup[]`
- `selected_items`
- `selected_new_items`
- `diagnostics`

这里需要保证：

- 同一新闻不会因多层命中而重复输出
- 主题优先级有单一决策点
- 所有跨层证据都能进入 `metadata / diagnostics`

#### 任务 8：隔离当前 `ai_filter` 存储细节

本轮不建议直接重写数据库 schema。

但建议至少新增一层 Selection 内部 repository / store facade，把这些能力包起来：

- active topic state
- analyzed ids
- persisted decisions

这样就能把：

- `selection` 的算法逻辑
- `storage` 的兼容细节

分开收口，避免以后一动 semantic / classifier 就碰底层 repo。

#### 任务 9：扩展 review / audit，支撑新五层管线审阅

Stage 4 的 outbox 需要继续保留，而且要更贴近新链路。

建议新增 / 调整产物：

- `stage4_snapshot.json`
- `stage4_selection_keyword.json`
- `stage4_selection_semantic.json`
- `stage4_selection_llm.json`
- `stage4_selection_review.md`
- `stage4_selection_audit.md`

重点不是“文件更多”，而是要让人能看出：

- keyword 命中了什么
- semantic 召回了什么
- 哪些样本进了 llm 仲裁
- 最后为什么保留 / 剔除

#### 任务 10：测试与验证顺序

建议固定验证顺序：

1. 定向单测
   - `tests/test_workflow_selection.py`
   - `tests/test_workflow_selection_ai.py`
   - 新增 semantic / classifier / arbitration 测试
2. 上下文级测试
   - `tests/test_context_selection.py`
3. stage4 审阅导出测试
   - `tests/test_workflow_selection_review.py`
   - `tests/test_workflow_selection_audit.py`
4. 阶段端到端
   - 只跑到 Selection
   - 生成 `outbox/` 审阅产物

### 6.9 这一版的推荐落地策略

如果按当前优先级推进，建议分 3 个小阶段落地，而不是一次性把 5 层全部做满。

#### Phase A：先把骨架搭对

先做：

- topic catalog
- semantic recall
- rerank hook
- unified projector

这一步完成后，Selection 就不再是“keyword / ai 两套平行系统”，而是一个统一骨架。

#### Phase B：再把 LLM 收缩成仲裁器

再做：

- llm arbitration prompt
- 小 batch 仲裁输入
- AI 结果投影改造

这一步完成后，第四层的 token 压力和噪声会明显下降。

#### Phase C：最后补第三层 classifier

最后做：

- classifier interface
- 训练数据导出
- baseline classifier
- 是否启用本机训练

这样可以把最不确定的本机训练部分放到最后处理，不阻塞主链路收口。

### 6.10 验收标准

第四环节重构完成后，至少应满足：

- Selection 形成统一五层结构，不再只是 keyword / ai 二选一
- 第二层 embedding 使用 `.env` 中同源 embedding 模型，API key/base 继承现有 provider
- reranker 具备明确接入口，但默认可关闭
- 第四层 LLM 不再对全量 100+ 标题做主分类，而只处理候选边界样本
- 第三层 classifier 不是硬依赖，机器性能不足时可关闭，但接口与数据沉淀路径已具备
- `SelectionService` 仍只对外输出原生 `SelectionResult`
- stage4 审阅产物能清楚展示 keyword / semantic / llm 的分层结果
- 相关测试与阶段端到端验证可以稳定跑通
