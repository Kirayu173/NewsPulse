## 3. 环节一：采集抓取

### 3.1 环节职责与边界

这是主链路的最前端环节，负责从配置指定的热榜源中抓取原始榜单内容，并把外部站点各自不同的返回格式，收敛为统一的“抓取层输出”。

这个环节只应该负责：

- 读取要抓哪些源
- 为每个源选择对应 handler
- 处理网络请求、重试、超时、代理
- 把每个源的返回结果规整成统一抓取结果
- 返回成功源、失败源、源名映射

这个环节不应该负责：

- 持久化到存储层
- 生成 `NewsData`
- 判断 report mode
- 计算新增标题
- 做 Selection / Insight / Render

也就是说，它的边界应该停在“抓到了什么”，不应该跨到“这些数据后面怎么用”。

### 3.2 当前功能模块

当前环节由以下模块组成。

#### A. 抓取源定义入口

- `config/config.yaml`
  - `platforms.enabled`
  - `platforms.sources`
- `newspulse/core/loader.py`
  - 负责把 YAML 中的 `platforms.sources` 读入运行时配置，落到 `config["PLATFORMS"]`
- `newspulse/context.py`
  - `AppContext.platforms` 暴露当前启用的平台列表

职责：

- 定义这次抓哪些热榜源
- 传给运行时编排层

#### B. 采集编排入口

- `newspulse/runner/news_runner.py`
  - `_crawl_data()`

职责：

- 从 `AppContext.platforms` 里取出平台
- 组装抓取用的 `ids`
- 调用 `DataFetcher.crawl_websites()`
- 接收抓取结果，交给后续环节

它当前仍是“抓取环节”和“后续归一化/入库环节”的交界点。

#### C. 抓取协调器

- `newspulse/crawler/fetcher.py`
  - `DataFetcher.fetch_data()`
  - `DataFetcher.crawl_websites()`

职责：

- 串行遍历每个 source
- 处理重试和随机等待
- 调用具体 source handler
- 汇总结果为统一抓取层输出

#### D. Source 注册中心

- `newspulse/crawler/sources/registry.py`
  - `SOURCE_REGISTRY`
  - `get_source_handler()`

职责：

- 把 `source_id` 映射到具体抓取函数
- 提供“source id -> handler”的分发机制

当前它是抓取环节里最核心的“源路由层”。

#### E. Source 基础客户端与通用工具

- `newspulse/crawler/sources/base.py`
  - `SourceClient`
  - `SourceItem`
  - HTML / RSS / URL / hash 等基础工具
- `newspulse/crawler/sources/common.py`
  - 文本清洗、通用 item 构造、feed 辅助函数

职责：

- 封装 requests session、代理、headers、timeout
- 给 source handler 提供统一网络访问能力
- 提供抓取后最小归一化对象 `SourceItem`

#### F. 各垂直 source handler 实现

- `newspulse/crawler/sources/mainland.py`
- `newspulse/crawler/sources/finance.py`
- `newspulse/crawler/sources/tech.py`
- `newspulse/crawler/sources/misc.py`

职责：

- 面向单一站点抓取
- 解析 JSON / HTML / RSS / 页面结构
- 输出 `list[SourceItem]`

这些 handler 才是真正“贴着外部网站协议”的地方。

### 3.3 当前输入数据

当前环节的输入主要有三类。

#### 输入 1：平台配置

来源：

- `config/config.yaml`
- `newspulse/core/loader.py`
- `newspulse/context.py`

当前运行时形状大致为：

```python
[
    {"id": "thepaper", "name": "澎湃"},
    {"id": "tencent-hot", "name": "腾讯热榜"},
    {"id": "cls-hot", "name": "财联社热榜"},
    {"id": "wallstreetcn-hot", "name": "华尔街见闻热榜"},
    {"id": "github-trending-today", "name": "GitHub Trending"},
    {"id": "hackernews", "name": "Hacker News"},
    {"id": "juejin", "name": "掘金"},
]
```

#### 输入 2：抓取运行参数

来源：

- `newspulse/runner/news_runner.py`
- `newspulse/context.py`

包括：

- `request_interval`
- `proxy_url`
- source 列表
- source alias / name

#### 输入 3：外部站点响应

来源：

- 各 source handler 访问的站点 API、HTML 页面、RSS feed

当前这是最不稳定的一层输入，不同站点的返回格式完全不同，因此都要在 handler 内部消化。

### 3.4 当前输出数据

当前这个环节最终输出 3 个对象：

1. `results`
2. `id_to_name`
3. `failed_ids`

其中 `results` 目前还是一个偏旧的 dict 形状：

```python
{
    "hackernews": {
        "OpenAI launches a new coding agent": {
            "ranks": [1],
            "url": "https://news.ycombinator.com/item?id=1",
            "mobileUrl": ""
        }
    }
}
```

`id_to_name` 形状：

```python
{
    "hackernews": "Hacker News",
    "github-trending-today": "GitHub Trending",
}
```

`failed_ids` 形状：

```python
["thepaper", "cls-hot"]
```

这一点很重要：

- source handler 的直接输出其实已经是 `list[SourceItem]`
- 但 `DataFetcher.crawl_websites()` 又把它重新折回了旧 dict 形状

这正是当前采集抓取环节最大的技术债之一。

### 3.5 数据从进来到出去的完整处理链路

下面按真实执行顺序说明。

#### 第 1 步：加载启用的平台列表

- `config/config.yaml` 中定义 `platforms.sources`
- `newspulse/core/loader.py` 把它写入 `config["PLATFORMS"]`
- `newspulse/context.py` 通过 `AppContext.platforms` 暴露给运行时

此时数据还是“配置对象列表”。

#### 第 2 步：运行编排层组装抓取请求

- `newspulse/runner/news_runner.py::_crawl_data()`

这里会把平台列表转换成：

- `source_id`
- 或 `(source_id, alias)` 二元组

得到一个给 `DataFetcher` 使用的 `ids` 列表。

#### 第 3 步：抓取协调器遍历 source

- `newspulse/crawler/fetcher.py::crawl_websites()`

行为包括：

- 遍历 `ids_list`
- 为每个 source 保留 `id_to_name`
- 调用 `fetch_data()`
- 相邻 source 之间做随机等待

此时系统开始真正触网。

#### 第 4 步：单个 source 执行抓取

- `newspulse/crawler/fetcher.py::fetch_data()`

行为包括：

- 根据 `source_id` 调 `get_source_handler()`
- 从 `SOURCE_REGISTRY` 中找到 handler
- 执行 handler
- 失败时进行重试

这一层的本质是“source 调度 + retry policy”。

#### 第 5 步：source handler 访问外部站点并解析

对应文件包括：

- `newspulse/crawler/sources/mainland.py`
- `newspulse/crawler/sources/finance.py`
- `newspulse/crawler/sources/tech.py`
- `newspulse/crawler/sources/misc.py`

这里会使用：

- `SourceClient.get_json()`
- `SourceClient.get_text()`
- `SourceClient.get_soup()`
- `SourceClient.get_feed()`

完成：

- 发请求
- 解析响应
- 提取标题 / URL / mobile URL
- 构造成 `SourceItem`

到这一步，单个站点的数据已经被规整成统一对象：

```python
SourceItem(
    title="OpenAI launches a new coding agent",
    url="https://news.ycombinator.com/item?id=1",
    mobile_url=""
)
```

#### 第 6 步：抓取协调器把 `SourceItem[]` 再折成旧汇总结果

- `newspulse/crawler/fetcher.py::crawl_websites()`

它会把 `list[SourceItem]` 转成：

- 以 `source_id` 为第一层 key
- 以 `title` 为第二层 key
- 值中记录 `ranks / url / mobileUrl`

并在同标题多次出现时，把排名累计到 `ranks` 列表。

这一层同时还会：

- 汇总 `id_to_name`
- 汇总 `failed_ids`

#### 第 7 步：返回给运行编排层

- `results`
- `id_to_name`
- `failed_ids`

返回点：

- `newspulse/runner/news_runner.py::_crawl_data()`

到这里，采集抓取环节结束。

后续下一环节“原始结果归一化 / 入库”会继续处理这些输出，把它们转换为 `NewsData` 并落库。

### 3.6 当前耦合点 / 脏点 / 旧形状残留

这一环节当前主要有 6 个问题。

#### 问题 1：抓取输出仍然回落到旧 dict 结构

source handler 已经产出 `SourceItem`，但 `DataFetcher.crawl_websites()` 又把它折回：

- `results[source_id][title]["ranks"]`
- `url`
- `mobileUrl`

这会让下游继续依赖旧结构，而不是直接吃原生抓取模型。

#### 问题 2：抓取协调器同时承担了“抓取”和“汇总归一化”

`DataFetcher` 现在既做：

- source 调度
- retry
- wait interval
- 聚合
- title 去重
- rank list 拼装

职责偏多。

#### 问题 3：source registry 是平铺硬编码表

`SOURCE_REGISTRY` 现在是一个大字典，虽然简单直接，但后续：

- source 元信息
- source 分类
- source 能力标签
- source 限流策略

都还没有结构化承载位置。

#### 问题 4：抓取层与运行编排层耦合较深

`NewsRunner._crawl_data()` 还在自己拼 source id/alias，并直接感知后续入库逻辑。

这说明“抓取任务定义”还没有独立成干净对象。

#### 问题 5：抓取失败信息过粗

当前 `failed_ids` 只有 source id，没有结构化失败原因、异常类型、响应信息。

这会限制：

- 观测性
- 重试诊断
- 后续失败源展示

#### 问题 6：缺少独立的采集层契约

当前采集环节没有像 `HotlistSnapshot`、`SelectionResult` 那样清晰的 stage contract。

这导致采集输出在进入存储层前，仍然是“临时拼出来的结构”。

### 3.7 本环节的精细化重构目标

后续这个环节应收口为：

```text
平台定义
-> 抓取任务
-> source handler
-> 原生抓取结果模型
```

目标包括：

1. 定义采集层原生输出模型，不再把 `SourceItem[]` 折回旧 dict
2. 把“单源抓取结果”“整批抓取结果”“失败结果”结构化
3. 把 retry / interval / source dispatch 与结果汇总拆开
4. 让 `runner` 不再感知抓取层内部拼装细节
5. 为下一环节“原始结果归一化 / 入库”提供稳定输入契约

建议未来目标形状至少包括：

- `FetchedSourceItem`
- `FetchedSourceBatch`
- `FetchFailure`
- `CrawlBatchResult`

### 3.8 建议拆分任务

建议把本环节再拆成以下子任务逐个做。

#### 任务 1：定义原生采集结果模型

目标：

- 替代 `results / id_to_name / failed_ids` 三件套

#### 任务 2：拆分 source dispatch 与 batch aggregation

目标：

- 让“抓单个源”和“汇总整批结果”成为两个清晰模块

#### 任务 3：把失败信息结构化

目标：

- 失败不再只有 source id
- 至少要包含 source_id、source_name、reason、exception_type

#### 任务 4：收紧 source registry 元信息

目标：

- 给每个 source 附带更稳定的定义对象，而不是只存函数指针

#### 任务 5：让 `runner` 只消费采集层契约

目标：

- `NewsRunner` 不再处理采集层内部数据拼接

### 3.9 验收标准

本环节完成时，应满足：

- 抓取层存在明确原生输出模型
- `DataFetcher` 不再输出旧 `results[source_id][title]` dict 结构
- source handler 统一输出稳定 item 契约
- 失败信息具备结构化字段
- `runner` 只消费“整批抓取结果对象”，而不是三件套临时结构
- 下一个环节“原始结果归一化 / 入库”直接消费新的采集层输出

### 3.10 本阶段实施范围

本阶段明确要做以下 5 件事，同时解决前面已经记录的主要问题：

1. 定义原生采集结果模型
2. 拆开 source dispatch 和 batch aggregation
3. 结构化失败信息
4. 收紧 source registry 元信息
5. 让 `runner` 只消费采集层契约，不再碰临时三件套

补充约束：

- 本阶段的重点是“采集层输出契约原生化”
- 不在本阶段重写整个 storage 仓储实现
- 但允许调整采集层到 storage 入口之间的转换函数签名
- 不允许继续让主链路生产代码依赖 `results / id_to_name / failed_ids` 三件套

### 3.11 代码现状分析结论

结合当前代码，采集环节的问题不是单点问题，而是 4 个问题缠在一起。

#### 结论 1：最核心的问题是“source handler 已原生化，但 fetcher 又把它降级了”

现在每个 source handler 的输出已经是：

- `SourceItem(title, url, mobile_url)`

这是一个很好的最小统一对象。

但 `newspulse/crawler/fetcher.py::crawl_websites()` 又把它折回：

- `results[source_id][title]["ranks"]`
- `url`
- `mobileUrl`

所以目前真正该改的不是 handler，而是：

- batch 输出契约
- fetcher 的职责边界
- runner 与 storage 的接缝

#### 结论 2：`crawl_websites()` 同时承担了三层职责

当前 `DataFetcher.crawl_websites()` 同时做了：

1. source 遍历与调度
2. source 抓取失败重试
3. 旧结构聚合与失败列表汇总

这导致它很难继续演进。

更合理的结构应该是：

```text
Source dispatch
-> per-source fetch result
-> batch result assembly
```

其中：

- dispatch 负责“调谁”
- fetch 负责“抓什么”
- batch assembly 负责“整批结果怎么组织”

#### 结论 3：结构化失败信息不能只在 crawler 内部改，必须考虑后续边界

当前失败信息只有：

- `failed_ids: list[str]`

而下游真正依赖失败信息的地方包括：

- `newspulse/storage/base.py`
- `newspulse/storage/repos/news.py`
- `newspulse/workflow/snapshot/service.py`
- `newspulse/workflow/render/html_page.py`
- `newspulse/workflow/render/notification_content.py`

因此合理方案不是只在 crawler 里加个异常对象就结束，而是：

- crawler 先输出结构化失败对象
- 采集到 storage 的转换入口至少先能理解结构化失败
- 之后由下一环节继续把失败信息完整存储和透传

#### 结论 4：source registry 现在的问题不是“能不能用”，而是“信息密度太低”

当前 registry 只保存：

- `source_id -> handler`

这足够运行，但不够支撑后续重构。

本阶段更合理的方向是让 registry 至少能表达：

- canonical id
- aliases
- default display name
- category / group
- handler

这样：

- 配置里可以继续写 alias
- 运行时可以统一到 canonical id
- 失败信息、日志、后续统计都可以更稳定

### 3.12 合理方案：目标结构

建议把采集环节收口成下面这条链路：

```text
平台配置
-> CrawlSourceSpec
-> SourceDefinition
-> SourceDispatcher
-> SourceFetchResult / SourceFetchFailure
-> CrawlBatchResult
-> convert_crawl_batch_to_news_data(...)
```

这里有几个关键设计决定。

#### 设计决定 A：保留 `SourceItem` 作为“handler 最小输出”，但不再让它直接充当整批抓取契约

原因：

- handler 只关心把站点数据解析成有序条目列表
- `SourceItem` 目前已经足够轻量稳定
- 真正需要升级的是“单源批次结果”和“整批抓取结果”

所以：

- `SourceItem` 可以继续保留在 `newspulse/crawler/sources/base.py`
- 但 batch 级别新增单独 dataclass

#### 设计决定 B：标题去重与 `ranks` 聚合从采集层退出

当前 `crawl_websites()` 会在抓取层就把同标题合并并累计 `ranks`

这不够合理，因为这其实已经是“原始结果归一化”的职责，应该属于下一环节。

因此本阶段建议：

- 采集层只保留“有序原始条目列表”
- 同标题聚合、`ranks` 生成、`rank=第一名次` 推导，交给“原始结果归一化 / 入库”环节

这会让两个环节边界更清楚。

#### 设计决定 C：失败对象先在采集层结构化，storage 入口先做消费

建议的新失败模型至少包含：

- `source_id`
- `source_name`
- `exception_type`
- `message`
- `attempts`
- `retryable`

如果本阶段不想立刻重写底层数据库表，可以先做到：

- 采集层输出结构化失败
- `convert_crawl_batch_to_news_data(...)` 暂时只把 `source_id` / `source_name` 映射进旧 `NewsData`

但这只能是“边界过渡”，不能再让 `runner` 继续拿三件套。

#### 设计决定 D：`runner` 的 `_crawl_data()` 改为只返回一个对象

当前：

```python
results, id_to_name, failed_ids = self.data_fetcher.crawl_websites(...)
```

应改成：

```python
crawl_batch = self.data_fetcher.crawl(...)
```

`runner` 只应做：

- 发起采集
- 把 `crawl_batch` 交给存储入口转换函数
- 不再理解内部三件套细节

### 3.13 推荐的数据模型方案

建议新增 `newspulse/crawler/models.py`，定义以下对象。

#### 1. `CrawlSourceSpec`

表示一次运行里“要抓哪个源”。

建议字段：

- `source_id: str`
- `source_name: str`

作用：

- 承接 `config["PLATFORMS"]`
- 作为 runner 传给 crawler 的稳定输入模型

#### 2. `SourceDefinition`

表示 registry 中的一个源定义。

建议字段：

- `canonical_id: str`
- `default_name: str`
- `aliases: tuple[str, ...]`
- `category: str`
- `handler: SourceHandler`

作用：

- 统一 alias
- 解决 registry 元信息过弱的问题

#### 3. `FetchedSourceItem`

如果想把 `SourceItem` 继续最小化，可以新增该对象；如果不想重复，也可以让它复用 `SourceItem`。

若新增，建议字段：

- `title`
- `url`
- `mobile_url`
- `position`

如果不新增，则位置由列表顺序隐式表达。

建议本阶段先不强行新增，直接沿用 `SourceItem`，避免 source handler 改动面过大。

#### 4. `SourceFetchResult`

表示单个 source 成功抓取结果。

建议字段：

- `source_id`
- `source_name`
- `resolved_source_id`
- `items: list[SourceItem]`
- `attempts`
- `metadata`

#### 5. `SourceFetchFailure`

表示单个 source 的结构化失败。

建议字段：

- `source_id`
- `source_name`
- `resolved_source_id`
- `attempts`
- `exception_type`
- `message`
- `metadata`

#### 6. `CrawlBatchResult`

表示整批采集输出，是本阶段最终主契约。

建议字段：

- `sources: list[SourceFetchResult]`
- `failures: list[SourceFetchFailure]`
- `metadata`

建议再提供便捷属性：

- `platform_names`
- `successful_source_ids`
- `failed_source_ids`

### 3.14 推荐的模块拆分方案

建议文件调整如下。

#### A. `newspulse/crawler/models.py`

新增：

- `CrawlSourceSpec`
- `SourceDefinition`
- `SourceFetchResult`
- `SourceFetchFailure`
- `CrawlBatchResult`

#### B. `newspulse/crawler/sources/registry.py`

从“字典映射”升级为“定义对象注册表”。

建议提供：

- `SOURCE_DEFINITIONS`
- `SOURCE_ALIAS_INDEX`
- `resolve_source_definition(source_id)`

同时保留兼容导出：

- `get_source_handler(source_id)`

但内部应改为先解析 definition 再返回 `handler`

#### C. `newspulse/crawler/fetcher.py`

保留 `DataFetcher`，但拆方法职责：

- `fetch_source(...)`：单源抓取 + retry，返回 `SourceFetchResult | SourceFetchFailure`
- `crawl(...)`：遍历 source spec 列表，汇总为 `CrawlBatchResult`

旧 `crawl_websites()` 不建议继续作为主入口保留。

如果必须临时保留，也只能：

- 作为测试或短期兼容包装
- 明确标记待删

#### D. `newspulse/runner/news_runner.py`

把 `_crawl_data()` 改成：

- 构建 `list[CrawlSourceSpec]`
- 调用 `self.data_fetcher.crawl(...)`
- 把 `CrawlBatchResult` 交给 storage 入口转换
- 返回 `CrawlBatchResult`

同时把 `_execute_mode_strategy()` 的签名收掉无用参数。

#### E. `newspulse/storage/base.py`

把：

- `convert_crawl_results_to_news_data(results, id_to_name, failed_ids, ...)`

替换为：

- `convert_crawl_batch_to_news_data(crawl_batch, crawl_time, crawl_date)`

这个函数在本阶段可以先承担“采集契约 -> 旧 NewsData”的边界转换职责。

注意：

- 这不等于完成了第 2 环节
- 它只是为了让第 1 环节的输出能进入现有存储层

### 3.15 推荐的实施步骤

建议按下面顺序落地，避免改一半难以收口。

#### 步骤 1：先立模型，不急着改 handler

先新增：

- `CrawlSourceSpec`
- `SourceDefinition`
- `SourceFetchResult`
- `SourceFetchFailure`
- `CrawlBatchResult`

目标：

- 先把主契约定下来

#### 步骤 2：升级 source registry

把 registry 改成 definition 驱动，但先保持：

- 旧 source id 都还能 resolve
- handler 本身不改签名

目标：

- 先把 source 元信息收紧

#### 步骤 3：重写 `DataFetcher` 主出口

新增主方法：

- `crawl(source_specs, request_interval)`

让它输出 `CrawlBatchResult`

目标：

- 主链路先摆脱三件套

#### 步骤 4：改 runner 消费新契约

`NewsRunner._crawl_data()` 只拿：

- `CrawlBatchResult`

并把 `_execute_mode_strategy()` 的无用抓取参数删除。

目标：

- 把“runner 不再碰三件套”先落实

#### 步骤 5：调整 storage 入口转换函数

新增：

- `convert_crawl_batch_to_news_data(...)`

必要时短期保留旧函数，但生产主链路应全部切到新函数。

目标：

- 主链路完成采集层原生契约切换

#### 步骤 6：补测试并删除旧采集出口

重点更新：

- `tests/test_builtin_sources.py`
- `tests/test_runner_news_runner.py`
- 与 `convert_crawl_results_to_news_data` 直接相关的测试

最后如果生产代码已无引用，应删除：

- `crawl_websites()` 或把它降为明确待删兼容包装
- 旧三件套断言测试

### 3.16 本阶段建议修改文件

建议涉及：

- `newspulse/crawler/models.py`（新增）
- `newspulse/crawler/fetcher.py`
- `newspulse/crawler/__init__.py`
- `newspulse/crawler/sources/registry.py`
- `newspulse/crawler/sources/__init__.py`
- `newspulse/runner/news_runner.py`
- `newspulse/storage/base.py`
- `tests/test_builtin_sources.py`
- `tests/test_runner_news_runner.py`
- 其他直接依赖旧转换函数的测试

### 3.17 风险与控制策略

本阶段最大的风险是“采集契约变了，但 storage / snapshot / 测试还没跟上”。

控制策略建议如下：

#### 风险 1：测试面较大

处理策略：

- 先保留 handler 级测试不动
- 先更新 fetcher / runner / storage conversion 相关测试

#### 风险 2：结构化失败信息会牵连后续环节

处理策略：

- 本阶段先让 crawler 输出结构化失败
- storage 边界先消费
- 第 2 环节再继续把失败信息完整透传到底层存储与 snapshot

#### 风险 3：source alias 切换可能影响已有配置

处理策略：

- registry 必须保留 alias 解析
- canonical id 统一在内部使用
- 外部配置允许继续写现有 alias

### 3.18 最终建议

这阶段最合理的方案不是“大改所有 source handler”，而是：

1. 保持 handler 签名不动
2. 在 batch 级别引入新契约
3. 用新 registry 元信息替代纯字典映射
4. 让 runner 先脱离三件套
5. 用一个新的 storage 入口转换函数承接边界

这样改动面集中、收益最大，而且能真正解决当前已经记录的问题，而不是只做表面命名调整。
